MelonEngine

A custom C++ game engine I am building from scratch on Vulkan 1.3. It is not a finished product — there is no shipped game yet — but the repository already has a runnable Sandbox host, a static Engine library, editor-style UI, and several graphics paths working end-to-end. This page highlights what exists today and what is still on the roadmap.

  • Vulkan 1.3
  • C++20
  • CMake
  • ImGui
  • VMA
  • vk-bootstrap

Status: active development — graphics and editor shell first; gameplay systems and content pipeline still expanding.

Sandbox in the editor

Repository layout

The public repo is organized for iteration, not polish: a thin Sandbox executable links against a static Engine target. Dependencies are pulled with CMake FetchContent (vk-bootstrap, Vulkan Memory Allocator, fastgltf) and vendored libraries (GLFW, Dear ImGui, Assimp).

  • Engine/ — Device, renderer, pipelines, descriptors, UI, core systems, math headers
  • Sandbox/ — Application loop, window polling, ImGui frame, render tick
  • Shaders/GLSL — gradient.comp, sky.comp, colored triangle/mesh stages (SPIR-V via compile script)
  • Vendor/ — GLFW, ImGui Vulkan backend, STB, Assimp binaries

Vulkan rendering (implemented)

Device & frame graph

  • Device — instance/surface/swapchain, frames in flight, VMA-backed AllocatedImage draw target, sync objects, and ImmediateSubmit for one-off GPU work.
  • Mesh upload path — index/vertex buffers uploaded through UploadMesh (rectangle primitive used in sandbox).
  • Validation-friendly setup — debug messenger and checked Vulkan calls (VK_CHECK) during bring-up.

Compute backgrounds

Pipelines builds separate gradient and sky compute pipelines from compiled .comp.spv modules. Effects are stored as ComputeEffect entries (pipeline handle, layout, push-constant data). The ImGui debug window switches the active effect and exposes live parameters — the starfield screenshot is the sky shader with tuned push constants.

Graphics pipelines

A PipelineBuilder helper configures dynamic rendering pipelines for colored triangle and mesh draws. The renderer composes background compute, geometry passes, and ImGui into each frame’s command buffer.

Editor UI (implemented)

  • ImGui + Vulkan — docking enabled; central viewport uses passthrough so the 3D/compute scene shows through the dock space.
  • Window registry — MelonImGui registers panels (menu, debug, project settings) with per-window open state.
  • Tilemap Editor — first-pass panel with save/load menu hooks wired to Tilemap core types (authoring workflow still early).
  • Compute Shaders panel — selects background effect index and edits data1 / parameter vectors at runtime.

Core systems (implemented / started)

  • WindowSystem + GLFW — event poll, swap, close handling; window title reports active Vulkan version.
  • ArenaAlloc — persistent allocation for application-level objects (renderer, window system) without scattered new in the sandbox entry path.
  • Input / Keycode — keyboard queries used in sandbox (e.g. UI theme toggle).
  • Debug / RuntimeAssert — logging and assert infrastructure for native debugging.
  • Math — header-only vec2/vec3/vec4, mat3/mat4, projection helpers under Engine/math/.

Still in progress

Comments and TODOs in the repo mark the next layers: fuller window-system integration, asset hot-reload, scene/entity model, physics, audio, and moving tilemap authoring from UI stub to persisted level data. fastgltf and Assimp are already linked for upcoming mesh import work.

What this demonstrates

  • Owning the full Vulkan bring-up — swapchain, allocators, pipelines, compute + graphics, and ImGui compositing in one frame loop.
  • Building engine ergonomics — deletion queues, pipeline builders, effect structs, and editor panels that share the same renderer state.
  • Shipping vertical slices inside an unfinished engine (gradient viewport → compute sky → docked tools) instead of waiting for a “complete” milestone.
